# Constellation：“控制-数据流解耦”的下一代零损耗文档解析引擎

## 一、 算法锐评：克制即是统治，解耦才是归途

在大模型（LLM）狂飙突进的时代，文档解析领域陷入了两个极端的泥潭：
1. **传统规则派（如 Mammoth, MarkItDown）**：极度依赖标准的样式标签（如 Heading 1）。一旦遇到业务人员“人肉排版”（用字号加粗伪造标题），瞬间退化为扁平的文本流，RAG 所需的拓扑结构彻底崩塌。
2. **端到端生成派（LLM 全文重写）**：试图把几十万字的文档丢给大模型强行输出 JSON。这带来了两个无法逾越的死穴：**极度高昂的算力成本**（输出 Token 与文档长度成正比，O(N) 账单），以及**不可根除的生成幻觉**（大模型肯定会篡改、漏字、替换同义词，这在法律、金融领域是灾难）。

**Constellation 的算法突破在于：在系统架构层实现了“语义控制流”与“物理数据流”的彻底解耦。**
- **LLM 只做控制中心**：在极高压缩率（~92.7%）的虚拟骨架上进行“降维打击”，只输出空间坐标（区间锚点）和层级判定。
- **FSM（确定性状态机）做数据传输**：根据 LLM 给出的坐标，辅以 **模糊锚定纠偏（MAP 极大后验估计）**、**字号交叉验证（物理特征回溯）** 以及最新加入的 **动态多层级前缀侦测（Prefix Detector）**，进行坐标对齐与直接提取。

**锐评**：Constellation 的设计体现了极高的工程美学——它不迷信大模型的全能，而是把 LLM 用在刀刃上（语义理解），把脏活累活丢给确定性算法。**大模型不再是打字员，而是打点器。** 这从架构物理维度上，将篡改率降到了绝对的 0，同时将千万字文档的 LLM 解析成本打到了接近边际成本为零的程度（降本近 80%~92.7%）。

## 二、 实际场景投射：互联网企业的“降维打击”

在互联网企业的实际业务中，Constellation 解决了以下几个极其致命的落地痛点：

### 1. 知识库与 RAG 系统的“幻觉断头台”
- **痛点**：企业内部文档（尤其是产品PRD、技术文档、运营SOP）格式野蛮生长。RAG 系统如果无法准确切分出第一章、第二章及其子节点，检索时的 Context 就会张冠李戴，导致 AI 客服天天胡说八道。
- **应用**：借助 Prefix Detector 对 10 大类非标符号（如 `1.`、`（一）`、`A.`、`➢` 甚至 Markdown 残留标记）的精准识别，Constellation 即使在没有任何样式属性的纯文本排版库中，也能精准重建层级树，让 RAG 切片命中率与逻辑关联性呈指数级上升。

### 2. 法务与金融审核的“绝对忠诚”
- **痛点**：法律合同、招股书、研报这种文档，错一个字、丢一个条款都是极其严重的责任事故。传统 OCR + LLM 会偷偷把“省略”写成“略”，或者漏掉某个金额条款。
- **应用**：Constellation 的提取动作是底层 FSM 截取的，跳过了大模型的生成黑盒。这意味着，**所见即所得，零字符修改**。系统只切分，不写字，100.00% 还原文档本来面目。对合规部门而言，这是唯一敢于上生产线的大规模重组方案。

### 3. 百万字级超长文档的“贫穷限制”
- **痛点**：把一本百万字的说明书灌给 GPT-4o 进行结构提取，光输出 Token（重写几十万字）的费用就高达几十甚至上百美金，且往往超时报错。
- **应用**：由于 Constellation 将 LLM 负担从 O(N) 压缩到了 O(K)（K 为标题数量）。在 `extreme_stress` 高压测试中，对于原本根本跑不通的千万级字符文档，Constellation 只需要不到 4,500 个输出 Token！**相当于把解析一个超大型项目的费用，从喝一杯星巴克的钱，降到了掉在地上的硬币。**

## 三、 结合论文与核心算法：为什么我们不可战胜？

根据我们《Constellation: Lossless Document Structuring via Control-Data Flow Decoupling》一文的详实论证，我们的技术护城河可以量化为以下几点：

### 1. 极致的性能压制（Token 经济学）
在面对一百万字符的极端压力测试时，传统端到端生成至少需要吐出约 16.8 万个 Output Token；而 Constellation 通过**滑动窗口切片 + 虚拟骨架传输 (RLE 压缩)**，只用了 **819 个 Token**，**输出成本骤降 99.51%，总消耗成本降低 79.9%**。在面对 1000 万字的极限挑战时，传统方案直接 OOM 崩溃，而 Constellation 依旧稳如磐石，实现 92.7% 的极致压缩省流，这在处理并发极高的大厂后台，意味着成百上千张 GPU 的节省。

### 2. 恐怖的特征自适应反击（物理特征回溯机制）
很多文档 LLM 会误判，比如出现了两个相同字号的“第二章”和“第三章”，但一个使用了缩进，由于 LLM 看不到字号，容易把第三章算作第二章的子标题。Constellation 引入了 **FSM 层级审计（Hierarchy Auditing）**，用确定的物理特征（Font Size）来制衡 LLM 的判断：
> *如果 LLM 说你要嵌套降级，但大盘扫了一眼你的物理字号和上面同级，或者你还有加粗，系统将粗暴且合理地将你连跳两级，强制并轨。*
在基准测试中，基于这种交叉验证，MarkItDown 遭遇极限非标场景准确率跌破 0，而 Constellation 依然保持 **1.0000 的绝对拓扑精确度**。

### 3. 多层级动态前缀侦测 (Multi-Tier Prefix Detector)
这是算法中最敏锐的嗅觉器官。传统文档解析通常写死一套正则去抓 `[1]`。Constellation 全新自研的前缀侦测系统，将侦测范畴跃升至 **十个优先级梯队**：
从复杂的“符合嵌套 (`1.2.3.4`) ”，到大厂喜闻乐见的“全角数字 (`（一）`) ”，再到法务独有的“官方编号 (`Article 5.`) ”。
算法在 P帧 (P-Frame) 进行 RLE 压缩折叠时，不是暴力截断，而是通过侦测前缀的长度，进行**动态窗口保留**。这保证了大模型在看到高度压缩骨架时，永远不会错失任何一个带有微弱标题潜质的“首句信标”，从根本上消灭了“标题被折叠池吞噬”的低级错误。

## 总结

Constellation 不是在用大模型“造轮子”，而是在给大模型“配一把精准的手术刀”。在当今这个算力昂贵、容错率低的商业狂潮中，Constellation 用最深刻的计算机科学第一性原理（控制与数据分离），为长文档的工业级 RAG 预处理，给出了真正意义上的终极解法。
